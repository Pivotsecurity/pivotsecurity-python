Multiple Authentication

“The only people who love usernames and passwords are hackers.”

The use of multiple authentication factors to prove one's identity is based on the premise that an unauthorized actor is unlikely to be able to supply the factors required for access. If, in an authentication attempt, at least one of the components is missing or supplied incorrectly, the user's identity is not established with sufficient certainty and access to the asset (e.g., a building, or data) being protected by multi-factor authentication then remains blocked.

How can you add early and reliably, yet providing a platform for future examination? Every IT manager dreads to ask these questions. They very well know, it would take months (if not years) to put a solution in place. You don’t have to with Pivot Security API.

Pivot Security API is developer friendly multi-factor authentication api. With Machine Intelligence, every attempt to compromise end user account will be carefully avoided by placing security on users hands SMS, EMAIL or APP baked multi-factor authentication driven by API. Never have to worry about balancing usability vs security.

API Reference

The Pivot Security API is organized around REST. Our API accepts form-encoded request bodies, returns JSON-encoded responses, and uses standard HTTP response codes, basic http authentication, and verbs.

You can use the Pivot Security API in test mode, which does not affect your live data or security restrictions (like DDOS flagging etc.). The API key you use to authenticate the request determines whether the request is live mode or test mode.

Authentication

The Pivot Security API uses API keys to authenticate requests. You can view and manage your API keys in the Pivot Security Dashboard.

All API requests must be made over HTTPS. Calls made over plain HTTP will fail. API requests without authentication will also fail.

Errors

Pivot Security API uses conventional HTTP response codes to indicate the success or failure of an API request. Each response will return with or without content based on state of error or request data.

Handling errors

Our API libraries raise exceptions for many reasons, such as a failed data validation, invalid parameters, authentication errors, and network unavailability. We recommend writing code that gracefully handles all possible API exceptions.

Metadata

Metadata is useful for sending additional, structured information on an object, especially when using API from your own application. If you are using our libraries, the metadata is already set for you, but if you choose, you can modify to suit your needs.

Metadata is crucial data to determine whether the customer is accessing as a end user or using automated scripts to target user authentication.

Create Customer

Creates a new, customer object. Verification channel can be email, sms or mobile APP.

Example request Sample curl request

Example request body

***Property Description ***
uid (Optional) customer ID
email (Optional) Customer email address
channel (required) Verification channel, SMS, EMAIL or APP

Example Success response

POST
/api/customer/create
curl -X POST https://api.pivtosecurity.com/api/customer/create -X POST -u '<PUBLIC-API-KEY>:' -d '{"email":"user@server.com", "channel":"EMAIL"}' -H "Accept: application/json"
{
  "email": "email@server.com", 
  "channel":"EMAIL"
}
{
  "status": "Success"
}
